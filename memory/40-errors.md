## 🔴 CRITICAL

### BUG-001: Утечка hostname в `make_msgid()`

| Поле | Значение |
|---|---|
| **ID** | BUG-001 |
| **Серьёзность** | 🔴 CRITICAL |
| **Локация** | `core/sender.py` — функция формирования `Message-ID` |
| **Обнаружен** | Аудит (задача 12) |
| **Описание** | Вызов `make_msgid()` без параметра `domain` использовал `socket.getfqdn()`, что вставляло реальный hostname машины отправителя в заголовок `Message-ID`. Это раскрывает инфраструктуру отправки и может использоваться для блокировки. |
| **Исправление** | Передан домен SMTP-аккаунта в параметр `domain`: `make_msgid(domain=smtp_account.domain)` |
| **Статус** | ✅ Исправлено |

> [!CAUTION]
> Заголовок `Message-ID` виден всем получателям и почтовым системам. Утечка hostname — критическая уязвимость приватности.

### BUG-002: Утечка SMTP-соединения при ошибке отправки

| Поле | Значение |
|---|---|
| **ID** | BUG-002 |
| **Серьёзность** | 🔴 CRITICAL |
| **Локация** | `core/sender.py` — блок отправки письма |
| **Обнаружен** | Аудит (задача 12) |
| **Описание** | Если `conn.sendmail()` выбрасывал исключение (таймаут, отказ сервера, ошибка данных), вызов `conn.quit()` пропускался. Соединение оставалось открытым → утечка сокетов → исчерпание ресурсов при массовой рассылке. |
| **Исправление** | Обёртка блока отправки в `try/finally` с гарантированным вызовом `conn.quit()` |
| **Статус** | ✅ Исправлено |

```diff
  conn = smtp_connect(account)
- conn.sendmail(from_addr, to_addrs, msg.as_string())
- conn.quit()
+ try:
+     conn.sendmail(from_addr, to_addrs, msg.as_string())
+ finally:
+     conn.quit()
```

## 🟡 SPAM

### BUG-003: X-Mailer заголовок — fingerprint массовой рассылки

| Поле | Значение |
|---|---|
| **ID** | BUG-003 |
| **Серьёзность** | 🟡 SPAM |
| **Локация** | `core/sender.py` — формирование MIME-заголовков |
| **Обнаружен** | Аудит (задача 12) |
| **Описание** | Статический заголовок `X-Mailer: CHARLY MAILER` во всех письмах позволял спам-фильтрам идентифицировать и группировать все сообщения как отправленные одной программой массовой рассылки. Это значительно повышало spam score. |
| **Исправление** | Рандомизация `X-Mailer` из пула реалистичных значений: `Mozilla Thunderbird 115.x`, `Apple Mail`, `Microsoft Outlook 16.0.xxxxx.xxxxx`, `The Bat! 11.x` и другие. Значение выбирается случайно для каждого письма. |
| **Статус** | ✅ Исправлено |

> [!TIP]
> Рандомизация X-Mailer — один из самых простых и эффективных способов снижения spam score. См. [[30-decisions]] для обоснования.

### BUG-004: CSV-экспорт без полей `had_cc` / `had_bcc`

| Поле | Значение |
|---|---|
| **ID** | BUG-004 |
| **Серьёзность** | 🟡 SPAM (низкий) |
| **Локация** | `core/stats.py` — функция CSV-экспорта |
| **Обнаружен** | Аудит (задача 12) |
| **Описание** | В CSV-отчёте статистики отсутствовали булевые поля `had_cc` и `had_bcc`. Без этих данных невозможно анализировать корреляцию между типом рассылки (прямая / CC / BCC) и показателями доставляемости. |
| **Исправление** | Добавлены колонки `had_cc` и `had_bcc` в CSV-экспорт |
| **Статус** | ✅ Исправлено |

## 🟢 MINOR

### BUG-005: Нет backoff при 4xx soft-bounce

| Поле | Значение |
|---|---|
| **ID** | BUG-005 |
| **Серьёзность** | 🟢 MINOR |
| **Локация** | `core/sender.py` — обработка SMTP-ошибок |
| **Обнаружен** | Аудит (задача 12) |
| **Описание** | При получении 4xx ответа от SMTP-сервера (temporary failure / soft-bounce) повторная попытка выполнялась немедленно, без задержки. Это могло привести к rate-limiting или полной блокировке IP/аккаунта сервером. |
| **Исправление** | Добавлен retry-sleep с экспоненциальной задержкой перед повтором после 4xx ошибки. Задержка реализована через `stop_event.wait()` для возможности прерывания (см. [[30-decisions]] — Event для stop/pause). |
| **Статус** | ✅ Исправлено |
