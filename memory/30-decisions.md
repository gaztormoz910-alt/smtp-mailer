> Почему выбраны конкретные подходы и технологии в SMTP MAILER. Каждое решение содержит контекст проблемы, выбранное решение и обоснование.

## 1. PySocks для SMTP через прокси

**Проблема:** Стандартный `smtplib` в Python не поддерживает подключение через прокси-серверы (SOCKS4/SOCKS5/HTTP).

**Решение:** Библиотека `PySocks` создаёт туннельный сокет через прокси, который передаётся в `smtplib.SMTP` как готовое соединение.

**Обоснование:**
- `smtplib` принимает параметр `socket` — можно подставить уже подключённый через прокси сокет
- PySocks поддерживает SOCKS4, SOCKS5 и HTTP CONNECT
- Не требует monkey-patching или модификации стандартной библиотеки

```python
import socks
sock = socks.socksocket()
sock.set_proxy(socks.SOCKS5, proxy_host, proxy_port)
sock.connect((smtp_host, smtp_port))
smtp = smtplib.SMTP()
smtp.sock = sock
```

## 2. `formataddr` для поля From

**Проблема:** Кириллические имена отправителя в заголовке `From:` могут вызывать спам-фильтры и некорректно отображаться.

**Решение:** Использование `email.utils.formataddr` с RFC 2047 кодированием.

**Обоснование:**
- RFC 2047 корректно кодирует Unicode-имена в заголовках (`=?utf-8?b?...?=`)
- Почтовые клиенты правильно декодируют и отображают имя
- Снижает вероятность попадания в спам по сравнению с raw UTF-8 в заголовке

```python
from email.utils import formataddr
from email.header import Header

formatted = formataddr((str(Header(display_name, 'utf-8')), email_address))
# Результат: =?utf-8?b?0JjQvNGP?= <user@example.com>
```

## 3. BCC только в envelope, не в заголовках

**Проблема:** Как добавить скрытую копию (BCC) так, чтобы получатели не видели адреса друг друга?

**Решение:** BCC-адреса включаются **только в envelope** (параметр `to_addrs` в `sendmail()`), но **не добавляются** в MIME-заголовки письма.

**Обоснование:**
- Протокол SMTP разделяет **envelope** (маршрутизация, `RCPT TO:`) и **заголовки** (видимые получателю)
- Если добавить BCC в заголовки — получатели увидят скрытых адресатов, что нарушает саму концепцию BCC
- `sendmail()` позволяет указать произвольный список получателей в envelope независимо от заголовков

> [!CAUTION]
> Никогда не добавляйте BCC-адреса в MIME-заголовки письма! Это раскроет «скрытых» получателей.

## 4. `threading.Event` для stop/pause

**Проблема:** Нужен механизм остановки и паузы рассылки без busy-waiting и с возможностью прервать задержку между письмами.

**Решение:** Два объекта `threading.Event`:
- `stop_event` — сигнал полной остановки
- `pause_event` — сигнал паузы (set = пауза)

**Обоснование:**
- `stop_event.wait(delay)` — **отменяемый sleep**: если событие установлено, wait() завершается мгновенно
- Нет busy-waiting (в отличие от `while not stopped: time.sleep(0.1)`)
- Мгновенная реакция на нажатие кнопки «Стоп» даже во время задержки
- Семантически чистый код: `if stop_event.is_set(): break`

```python
# Вместо time.sleep(delay):
if stop_event.wait(timeout=delay):
    break  # Остановка запрошена — выходим немедленно
```

См. [[10-architecture]] — раздел «Модель многопоточности».

## 5. `sendmail()` вместо `send_message()`

**Проблема:** `send_message()` автоматически извлекает получателей из заголовков `To`, `Cc`, `Bcc` и **удаляет** Bcc из заголовков.

**Решение:** Использование низкоуровневого `sendmail(from_addr, to_addrs, msg.as_string())`.

**Обоснование:**
- `send_message()` не позволяет точно контролировать envelope-получателей
- При BCC-only рассылке `send_message()` может некорректно сформировать список получателей
- `sendmail()` даёт полный контроль: envelope-отправитель, envelope-получатели и тело письма — три независимых параметра

> [!IMPORTANT]
> `sendmail()` — единственный способ гарантировать, что BCC работает правильно при массовой рассылке.

## 6. Режим `consistent_links`

**Проблема:** Если макрос `[[LINK]]` заменяется случайной ссылкой при каждом вхождении, спам-фильтры обнаруживают паттерн «случайных URL» в теле письма.

**Решение:** Режим `consistent_links` — **одна и та же** ссылка для всех `[[LINK]]` в пределах одного письма.

**Обоснование:**
- Реальные письма обычно содержат одну и ту же ссылку несколько раз (кнопка + текстовая ссылка)
- Разные случайные URL в одном письме — явный признак автоматической генерации
- Между письмами ссылки по-прежнему ротируются для распределения нагрузки

## 7. Singleton `JsonLogger`

**Проблема:** Несколько потоков пишут логи одновременно → риск повреждения файла.

**Решение:** Единственный экземпляр `JsonLogger` с одним `write_lock`.

**Обоснование:**
- Один `threading.Lock` гарантирует атомарность записи
- Один открытый файловый дескриптор — нет конфликтов при flush
- Простота использования: `logger.log(event, data)` из любого потока

> [!NOTE]
> Singleton реализован через модульную переменную, а не через метакласс — проще и достаточно для данного случая.

## 8. Рекурсивный спинтакс через `while`-цикл

**Проблема:** Спинтакс может быть вложенным на произвольную глубину: `{a|{b|{c|d}}}`. Рекурсивная функция может упереться в лимит стека Python (`RecursionError`).

**Решение:** Итеративная обработка через `while re.search(pattern, text)` — обрабатывает самый глубокий уровень на каждой итерации.

**Обоснование:**
- Python ограничивает глубину рекурсии (по умолчанию 1000)
- Итеративный подход: на каждой итерации `re.sub` заменяет **самый внутренний** `{...|...}` → постепенно «разворачивает» вложенность
- Нет ограничения на глубину вложенности
- Проще для отладки

```python
while re.search(r'\{[^{}]+\}', text):
    text = re.sub(r'\{([^{}]+)\}', lambda m: random.choice(m.group(1).split('|')), text)
```
